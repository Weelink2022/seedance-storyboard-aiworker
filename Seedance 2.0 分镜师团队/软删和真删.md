## 1. 资源 (resource) ↔ 集合 (collection) 是多对多关联

### 数据模型

```
resource (头等实体,resource 表)
   │
   │ 通过 collection_resource (collection, resource, ...) 多对多挂载
   │
collection (容器,collection 表)
```

`collection_resource` 表字段:`collection`, `resource`, `date_added`, `comment`, `rating`, `use_as_theme_thumbnail`, `sortorder` — **没有任何外键约束**。

### 一个资源可以同时挂载在 N 个 collection

实测 shot resource `ref=100900`:

```
collection=10084 镜头 001       date_added=2026-04-19 14:21:08
collection=10083 第1场          date_added=2026-04-19 14:21:08
collection=10066 测试漫剧        date_added=2026-04-19 14:21:09
collection=10076 第1集《开篇》    date_added=2026-04-19 14:21:09
collection=10077 分镜库          date_added=2026-04-19 14:21:09
```

同一个 resource 出现 **5 行 collection_resource 映射**,资源本身**不复制**。

### 跟手工「加入活跃收藏夹」是同一份代码

| 路径 | 入口 | 调用 |
|---|---|---|
| 用户手工「加入活跃收藏夹」 | 浏览器搜索结果勾选 → 按钮 | `add_resource_to_collection($resource, $collection)` |
| ai_series 自动挂载 | OCF 流程 / shot 创建 / scene 创建 | `ai_series_collection_add_resource()` → 内部调 `add_resource_to_collection(..., true)` |

两条路径在 [collections_functions.php:465](/var/www/html/resourcespace/10.7/include/collections_functions.php#L465) 汇聚到同一句 SQL:

```sql
INSERT INTO collection_resource(collection, resource, sortorder) VALUES (?, ?, ?)
```

差异只在 `$smartadd` 参数:`true` 绕过写权限检查(异步 / hook 路径),`false` 要求 collection_writeable。

## 2. 资源没有「根 collection」概念

**RS 数据模型里资源不属于任何 collection** — 资源和 collection 是平等的对等实体,通过 `collection_resource` 多对多关联。

### 证据

| 维度 | 实测 |
|---|---|
| `resource` 表 65 个字段是否有 `origin_collection` 类字段 | **没有** |
| `collection_resource` 表是否标记「原始挂载」 | **没有** — 所有挂载行字段对称 |
| resource 100900 时序 | 18 18:48:21 创建,19 14:03:03 首次挂 collection — **中间 19.5 小时游离 0 个 collection** |

### "根" 的伪概念 — 容易混淆的两个其他字段

1. **资源所有权**:`resource.created_by`(指向 user,不是 collection)
2. **业务归属**(ai_series 自己加的一层):`ai_series_tree.collection_ref` + `company_ref`,通过 tree 反推

### 找资源「最早 collection」的方法

```sql
-- 沿 collection_log 找最早被挂的 collection(可追溯,非强约束)
SELECT collection, date FROM collection_log
 WHERE resource = ? AND type = 'a'
 ORDER BY date, ref LIMIT 1;
```

## 3. 用户「不在 collection 里创建资源」

### 核心认知

RS 的资源创建和 collection 是**两个独立动作**。UI 上看起来「上传到收藏夹」是把两步合一,但底层是:

1. `create_resource()` 进 `resource` 表 — **不涉及 collection**
2. `add_resource_to_collection()` 挂到用户指定的 collection(可选)

### 证据 — `create_resource()` 签名

[resource_functions.php:567](/var/www/html/resourcespace/10.7/include/resource_functions.php#L567):

```php
create_resource($resource_type, $archive, $user, $origin, $file_extension, $run_macro)
                ↑              ↑       ↑     ↑       ↑                  ↑
            资源类型         状态    用户   来源    扩展名             宏
```

**参数列表完全没有 collection**。SQL:

```sql
INSERT INTO resource(resource_type, creation_date, archive, created_by, file_extension, no_file)
VALUES (?, NOW(), ?, ?, ?, 1)
```

### UI 上传路径(印证)

[upload_batch.php:708-741](/var/www/html/resourcespace/10.7/pages/upload_batch.php#L708-L741):

```php
$ref = create_resource($resource_type, $setarchivestate, -1, $lang["createdfromwebuploader"]);
// ↑ 第 1 步:资源已存在

if (is_numeric($collection_add)) {
    add_resource_to_collection($ref, $collection_add, false, "", $resource_type);
    //      ↑ 第 2 步:挂到用户指定的 collection
}
if ($upload_then_edit && ...) {
    add_resource_to_collection($ref, $upload_review_col, ...);
    //      ↑ 第 3 步:额外挂到"我的新上传"(用户专属 review collection)
}
```

### 三种 UI 场景

| 场景 | 资源在哪 | collection 挂载 |
|---|---|---|
| 从「我的收藏夹」页面点上传 | resource 表 | 挂这个收藏夹 + 挂"我的新上传" |
| 从全局上传按钮 | resource 表 | 仅挂"我的新上传" |
| 用户的 `$usercollection` 默认 0 | resource 表 | **0 个 collection 挂载**(完全游离) |

### ai_series 实体的特殊性

`character/scene/prop/voice` 实体(resource_type=17/18/19/21)**甚至可以连 collection 都不挂** — 通过 `ai_series_project_asset` 关联到项目,`resource_node` 存 extra_config 元数据,跟 collection 完全脱钩。

## 4. 删除按钮:软删和真删共用同一函数

### 同一入口,两个分支

[delete_resource() resource_functions.php:2861](/var/www/html/resourcespace/10.7/include/resource_functions.php#L2861):

```php
function delete_resource($ref) {
    // 守门员 — 一次性鉴权
    if (!$resource ||
        ((checkperm("D") || !get_edit_access(...) || lock_user 检查)
         && !hook('check_single_delete') && PHP_SAPI != 'cli')) {
        return false;
    }

    $current_state = $resource['archive'];

    global $resource_deletion_state;  // 当前机器 = 3
    if (isset($resource_deletion_state) && $current_state != $resource_deletion_state) {
        // ───────── 软删分支 ─────────
        update_archive_status($ref, $resource_deletion_state, $current_state);
        // 把 archive 字段改成 3,然后 return true
        return true;
    }

    // ───────── 真删分支(已是 archive=3) ─────────
    unlink(磁盘文件)              // 文件
    clear_resource_data($ref)     // resource_dimensions / resource_related / resource_node
    DELETE FROM resource WHERE ref=?
    DELETE FROM collection_resource WHERE resource=?
    DELETE FROM resource_custom_access WHERE resource=?
    DELETE FROM external_access_keys WHERE resource=?
    DELETE FROM resource_alt_files WHERE resource=?
    DELETE FROM annotation_node + annotation
}
```

### 按钮文案在 UI 层切换

[view.php:1209](/var/www/html/resourcespace/10.7/pages/view.php#L1209):

```php
$deletetext = (isset($resource_deletion_state) && $resource["archive"] == $resource_deletion_state)
    ? $lang["action-delete_permanently"]   // 资源已是 archive=3 → "永久删除"
    : $lang["action-delete"];               // 资源还活着 → "删除"
```

### archive 状态码

| 值 | 含义 |
|---|---|
| -2 | pending review(待审) |
| -1 | pending submission(草稿) |
| 0 | active(正常可见) |
| 1 | pending archive(待归档) |
| 2 | archived(已归档) |
| **3** | **deleted(回收站)= $resource_deletion_state** |

### 本机实际分布

```
archive=-2 → 112 个资源
archive=-1 → 2 个资源
archive= 0 → 2565 个资源
archive= 2 → 424 个资源
archive= 3 → 1157 个资源 (回收站,等待管理员清空)
archive=999 → 56 个资源
```

## 5. 软删时挂载会消失(默认配置)

### 关键配置

```
$resource_deletion_state = 3                          (默认)
$remove_deleted_resources_from_collections = true     (默认)
```

### 软删触发的副作用

[resource_functions.php:6300-6309](/var/www/html/resourcespace/10.7/include/resource_functions.php#L6300-L6309):

```php
if ($remove_deleted_resources_from_collections && $resource_deletion_state == $archive) {
    foreach ($resource as $ref) {
        $in_collections = ps_query('SELECT collection, resource FROM collection_resource WHERE resource = ?', ['i', $ref]);
        ps_query('DELETE FROM collection_resource WHERE resource = ?', ['i', $ref]);
        //  ↑ 即使是软删,挂载也会被全部清掉!
        foreach ($in_collections as $col) {
            collection_log($col['collection'], LOG_CODE_COLLECTION_DELETED_RESOURCE, $col['resource']);
        }
    }
}
```

### 结论

**第一次点删除(进 archive=3)的瞬间**,`collection_resource` 表里该资源的所有行**全部消失**。

资源同时在所有 collection 视图里隐形(其实是行已删,不是"被过滤"),但 `resource` 行仍在 archive=3 状态。

### 但仍有"漏网鱼"(2026-05-23 发现,2026-05-24 §51b 清完)

§51 主修复后审计发现:

```
archive=3 资源总数: 1157
  其中还有挂载的: 73 个
  总挂载行数: 162
```

73 个软删资源仍有挂载 — 早期某段已删代码直接 `UPDATE resource SET archive=3`(没走 `update_archive_status()`)产生的遗留。

分类:
- Script (type=16) "导演讲戏本/分镜镜头提示词(N)" 历史版本系列 44 个 = 88 挂载行
- Photo (type=1) 实体输出图 19 个 = 36 行
- Shot (type=15) 8 个 = 32 行
- Pt (type=2) 2 个 = 6 行

**不跨租户**(挂在 ai_series_tree 管理的 collection 上,被 `NOT IN` 排除),但**用户在自己项目里能看到历史版本污染**(如剧本库出现 13 个 "导演讲戏本" 版本)。

**§51b 已清**:`/tmp/cleanup_archive3_leaks_section51b.php` — 删 162 行 collection_resource + 写 162 行 collection_log type='d'(审计)。验证:剩余 0 漏网。详见 [漫剧Agents.md §51b](/root/Seedance%202.0%20分镜师团队/漫剧Agents.md)。

## 6. 挂载不阻拦真删

### `collection_resource` 没有外键

实测:

```
information_schema.KEY_COLUMN_USAGE WHERE table='collection_resource'
→ 无任何外键
```

### 真删时同步清挂载

[resource_functions.php:2956](/var/www/html/resourcespace/10.7/include/resource_functions.php#L2956):

```sql
DELETE FROM collection_resource WHERE resource = ?  -- 同事务清所有挂载
```

### 唯一阻拦真删的因素(都跟挂载无关)

[resource_functions.php:2867-2884](/var/www/html/resourcespace/10.7/include/resource_functions.php#L2867-L2884):

1. 资源不存在
2. 当前用户被设了 `D` 权限(禁止删除)
3. `get_edit_access()` 返回 false(没编辑权限)
4. 资源被别人锁了(`lock_user`)
5. 文件标 readonly(FStemplate)
6. 正在转码

**有 collection 挂载完全不影响**。

## 7. 谁能软删 / 谁能真删 — 同一套权限,跟 owner 没必然关系

### 关键认知

**软删 和 真删 走同一个 `delete_resource()`,守门员只检一次**。差异只是资源当前 archive 状态决定走哪个分支。

所以"谁能软删"和"谁能真删"问的是**同一个权限问题**:用户对该 archive 状态有没有编辑权。

### `get_edit_access()` 决策树

[resource_functions.php:5357](/var/www/html/resourcespace/10.7/include/resource_functions.php#L5357),由上到下短路:

1. **`customediteaccess` 钩子显式说能** → ✅
2. **`$edit_access_for_contributor=true` + 是 owner** → ✅
3. **`$edit_only_own_contributions=true` + 不是 owner** → ❌
4. **没有 `e<status>` 或 `ert<resource_type>` 权限** → ❌
5. **`z<status>` 阻止权限存在** → ❌
6. **status<0 且不是 admin('t')也不是 owner** → ❌
7. **edit_filter 不匹配该资源** → ❌
8. **通过** → ✅

### 本机配置 — owner 没有自动特权!

```
$edit_access_for_contributor = false
$edit_only_own_contributions = false
```

意味着 **owner 跟其他人遵守同一套权限规则** — owner 本身没有 RS 层面的「我的资源我做主」特权。

### 本机 LEAD 组实际权限

`TPL_LEAD (43)` 和 `TPL_OPS_LEAD (45)` permissions 串一样:

```
v,s,e-1,e-2,e0,g,c,q,f*,j*,z3,t,u,U,wf2,wf3,wf4,ac,acw,acg,acr,aca,act,acs,ace,acd
```

关键 token:
- `e-2 / e-1 / e0` — 能编辑 archive=-2/-1/0 的资源
- **没有** `e1 / e2 / e3`
- **`z3`** — **阻止编辑** archive=3 的资源!

### 本机 LEAD 用户的实际能力

| 资源当前 archive | LEAD 能做什么 |
|---|---|
| -2 (待审) | ✅ 软删 → 进 3 |
| -1 (草稿) | ✅ 软删 → 进 3 |
| 0 (正常) | ✅ 软删 → 进 3 |
| 1 (待归档) | ❌ 没 e1,挡 |
| 2 (已归档) | ❌ 没 e2,挡 |
| **3 (回收站)** | ❌ **真删不了!** `z3` 挡死 |

**LEAD 等于「只能扔进回收站」**,真删要 admin 来做。这是 RS 的设计意图:就像现实公司,员工能扔东西进废纸篓,但销毁是 IT 的事。

### 不同角色的删除能力总览

| 角色 | 软删 archive=0 | 真删 archive=3 |
|---|---|---|
| **admin (user 1)** | ✅ 有 `t` + 全权 | ✅ 同 |
| **TPL_LEAD / TPL_OPS_LEAD** | ✅(有 e0) | ❌(z3 挡) |
| **TPL_EDITOR / VIEWER** | 取决于 group permissions | 通常 ❌ |
| **资源 owner 但非 admin** | **无特权** — 看 group permissions(本机 `$edit_access_for_contributor=false`) | 同上 |
| **被设 'D' 权限** | ❌ 直接挡 | ❌ 同挡 |
| **资源被锁 (`lock_user`)** | ❌ 锁定者外谁都不能 | ❌ 同 |
| **CLI 任何脚本** | ✅ 总是放行 | ✅ 同 |

## 8. 关键架构含义

### 8.1 资源是头等实体,collection 是索引

- 不是「资源住在 collection 里」
- 资源可以游离 0 个 collection 存在 — `search.php` 仍能搜到它
- 删 collection 不删资源(§51 helper 这样设计)
- 没有「根 collection」概念

### 8.2 软删 vs 真删 vs 移除挂载 — 三个独立操作

| 操作 | 影响 resource 表 | 影响 collection_resource 表 | 触发条件 |
|---|---|---|---|
| **挂到 collection** | ❌ 不动 | ➕ INSERT 一行 | `add_resource_to_collection()` |
| **从 collection 移除**(不是删除) | ❌ 不动 | ➖ DELETE 这一对挂载 | `remove_resource_from_collection()` |
| **软删**(第一次点) | ✏️ archive=3 | ➖ DELETE WHERE resource=? (清全部挂载) | `delete_resource()` 走软删分支 |
| **真删**(第二次点) | ❌ DELETE 整行 | ➖ DELETE WHERE resource=? (清全部挂载) | `delete_resource()` 走真删分支(需 admin) |

### 8.3 业务层 vs 权限层的隔离

`ai_series_tree_delete_node()` (§51) 用裸 `ps_query()` DELETE,**绕过 RS 权限层**。这是「应用层内部清理」,合法调用方触发(reset / cleanup / delete_episode)就执行,不依赖用户身份。

跟 view.php 用户点删除是两套世界:
- view.php 走 `delete_resource()` → RS 权限检查
- ai_series helper 走 `ps_query()` → 跳过权限层

§51 修复后能彻底解决跨租户:**应用层删干净了,不依赖 RS 权限**。

### 8.4 跟 §51 修复的呼应

```php
// ai_series_tree_delete_node — 只动 collection 和 tree,不动 resource
DELETE FROM collection_resource WHERE collection = ?
DELETE FROM collection         WHERE ref = ?
DELETE FROM ai_series_tree     WHERE ref = ?
// ❌ resource 表完全不动
```

资源是否被「丢失」取决于它**是否还挂在别的 collection 上**:
- 仍挂在别处 → 别处 collection 视图照样看到
- 只挂在被删的 collection → 变「游离资源」,RS 主搜索屏仍能搜到(`search.php` 不需要 collection)

## 9. 一句话总结

**RS 的资源模型是平等多挂(像 Gmail label,不是文件夹)。删除分支由资源当前 archive 决定,删除权限由用户对该 archive 状态的编辑权决定,跟 owner 身份在本机配置下无关。本机 LEAD 用户只能软删 active 资源 → 扔进 archive=3 回收站;真删(包括清空回收站)只有 admin 能做。**
