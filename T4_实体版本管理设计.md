# T4 — 实体版本管理 设计文档

> 状态:**设计待审** · 2026-05-16
> 范围:character / scene / prop / voice 四类实体的"多版本并存 + 用户选用"机制

---

## 1. 要解决的问题

### 1.1 用户实际诉求(用户原话)
> "老的不删,新的新建,做好关联,用户选最终使用哪个局部版本。"

把"重跑/重做"从**破坏式替换**改成**叠加式版本**:
- 对某个实体(如"纽约曼哈顿办公室"场景)不满意 → 换个模型 / 重新生成 → **产生一个新版本**
- 老版本**保留不删**
- 用户在 UI 上对比、挑选 → 选中的版本成为 `active`
- binding / shot 引用该实体时,自动指向 `active` 版本

### 1.2 当前痛点(实测)
| 现象 | 根因 |
|---|---|
| 重跑 OCF,LLM 命名漂移("研究报告" vs "Dr. Miller 的研究报告")→ 建出重复实体 | fixed_id = 归一化名 sha1,名字一漂移就是新实体 |
| 重跑想换某个场景的图,只能整体重来 | 没有"单实体重做"入口 |
| 不满意某张图,没法保留旧的同时试新的 | 实体与图是 1:1 死绑 |

### 1.3 不做什么(范围边界)
- **不改 OCF 主流程**:一键成片默认仍是"每实体 1 个版本(v1)"
- 多版本是**用户手动触发**的高级功能,不自动多模型生成(省 LLM)
- voice 实体也纳入版本管理(跟 character/scene/prop 同等)

## 2. 数据模型

### 2.1 核心概念

```
logical_group (业务实体, 如"纽约曼哈顿办公室")
   ├── version 1  → RS resource A (entity_ref=103143)  [is_active=1]
   ├── version 2  → RS resource B (entity_ref=105001)  [is_active=0]
   └── version 3  → RS resource C (entity_ref=105050)  [is_active=0]
```

- 一个 `logical_group` = 用户眼里的"一个实体"
- 每个 `version` = 一条真实 RS resource(rtype 17/18/19/21)
- 同 group 同一时刻**只有一个** `is_active=1`

### 2.2 不新建表 — 复用 EXTRA_CONFIG

每类实体都有 EXTRA_CONFIG 字段(character 154 / scene 略 / voice 168 / prop 略)。
版本元信息存进 EXTRA_CONFIG JSON,新增 4 个键:

| 键 | 含义 | 示例 |
|---|---|---|
| `logical_group` | 业务实体稳定 ID(同 group 多版本共享) | `LG-scene-7f3a2b` |
| `version_id` | 版本序号 | `1` / `2` / `3` |
| `is_active` | 是否当前选用版本 | `true` / `false` |
| `source_model` | 这个版本用什么模型生成的 | `gemini-3-flash-preview` / `deepseek-chat` |

**为什么不新建表**:RS 资源体系已有 EXTRA_CONFIG,加表要管外键/迁移/一致性。EXTRA_CONFIG 加键最小侵入。

### 2.3 logical_group 怎么定 — 关键设计

不能用"名字 sha1"(名字会漂移)。三种候选:

| 方案 | logical_group 来源 | 评价 |
|---|---|---|
| **A. director 清单序号** | director-analysis.md 的人物/场景/道具清单里,每项的**出现顺序** → `LG-scene-1`...`LG-scene-8` | 稳定(剧本不变,清单顺序不变);但用户改剧本后序号会错位 |
| **B. 首次创建时分配 + 后续匹配** | 第一次建实体时生成 `LG-xxx`(随机);重跑时用**名字模糊匹配**找回同 group | 容忍命名漂移;但模糊匹配可能错配 |
| **C. 用户手动归并** | 系统按名字 sha1 建,UI 上让用户把"研究报告"和"Dr.Miller的研究报告"手动标成同 group | 100% 准;但要人工 |

**建议:A 为主 + C 兜底**
- OCF 建实体时,按 director 清单顺序分配 `logical_group = LG-<type>-<序号>`
- 同一个清单项重跑 → 同序号 → 同 group → 自动归为新版本
- 万一漂移/错位,UI 提供"手动归并/拆分 group"作兜底

### 2.4 binding token 怎么跟随 active 版本

binding 的 prompt_text 里 token 是 `[[AI_BINDING_ASSET|scene|<rid>|<photos>]]`,`<rid>` 是具体 resource ref。

**方案 X(token 写 group,运行时解析)**:
`[[AI_BINDING_ASSET|scene|LG-scene-1|]]` — 送 Seedance 前,实时查 LG-scene-1 的 active 版本 rid。
- 优点:切版本零成本(binding 不用动)
- 缺点:token 格式变,前端 chip 渲染、§38.26 都要改

**方案 Y(token 写 rid,切换时批量改写)**:
token 还是写具体 rid。用户切 active 版本时,后端扫所有 binding,把该 group 旧 rid 换成新 rid。
- 优点:token 格式不变,改动小
- 缺点:每次切换一次批量 UPDATE binding

**建议:方案 Y** — token 格式不变(§38.26/前端 chip 不用动),切换时批改 binding。切换是低频操作,批改可接受。
但**只批改 draft/complete 状态的 binding**(submitted/success 的不动,避免改用户已提交的)。

## 3. UI 设计

### 3.1 数字资产屏 — 实体卡片加版本切换

现状:数字资产屏每个实体一张卡片(缩略图 + 名字)。

改造:同 logical_group 的多版本**折叠成一张卡**,卡上加版本切换控件:

```
┌─────────────────────────────────────┐
│  纽约曼哈顿办公室                      │
│  ┌────┐                              │
│  │ 图 │   [v1 gemini ✓] [v2 deepseek]│  ← 版本 tab,✓=active
│  └────┘   [+ 新版本]                  │
│                                       │
│  当前版本: v1 (gemini-3-flash-preview) │
│  创建于 2026-05-16 12:17              │
└─────────────────────────────────────┘
```

- 版本 tab:点击切换查看;`✓` 标记 active
- hover 某 tab → 预览那个版本的图/提示词
- 点 `[设为当前]` → 该版本变 active(触发 binding 批改)
- `[+ 新版本]` → 弹模型选择 → 触发后端"局部重做"

### 3.2 "+ 新版本" 弹窗

```
┌─── 为「纽约曼哈顿办公室」生成新版本 ───┐
│  选择模型:                            │
│   ( ) gemini-3-flash-preview          │
│   (•) deepseek-chat                   │
│   ( ) doubao-seed-2-0-pro             │
│                                       │
│  说明: 用新模型重新生成该场景的提示词  │
│        + 出图。老版本保留。            │
│                                       │
│         [取消]      [开始生成]         │
└───────────────────────────────────────┘
```

### 3.3 版本对比(可选,二期)

并排看两个版本的图 + 提示词 diff。一期可不做,先有切换即可。

## 4. 后端接口

### 4.1 新增 API(api.php)

| action | 作用 |
|---|---|
| `entity_versions` | GET:返回某 logical_group 的所有版本列表 |
| `entity_set_active` | POST:把某版本设为 active + 批改引用它的 binding |
| `entity_new_version` | POST:对某 logical_group 用指定模型生成新版本(触发局部重做) |

### 4.2 "局部重做" 后端逻辑(entity_new_version)

最难的一块。对单个实体重做,不跑整个 OCF:

```
1. 拿到 logical_group + 目标模型
2. 找该 group 的清单项(从 director-analysis.md 对应行)
3. 只调一次 LLM(art-designer,单实体 prompt)生成新提示词
4. 建新 RS resource(version_id+1, is_active=0, source_model=目标模型)
5. 出图(character/scene/prop)/ 存 profile(voice)
6. 不动 binding(用户手动"设为当前"才批改)
```

**简化**:一期可以**不做"单实体局部重做"** — `[+ 新版本]` 先只支持"上传本地文件作为新版本"(用户自己拿图)。"换模型自动重做"放二期。
→ 一期 = 多版本**容器** + 切换;二期 = 自动重做。

## 5. 工作量评估 + 分期

### 一期(版本容器 + 切换)— 约 1-1.5 天
| 子任务 | 内容 |
|---|---|
| T4.1 | OCF 建实体时写 logical_group / version_id / is_active / source_model 进 EXTRA_CONFIG |
| T4.2 | asset_pool API 按 logical_group 聚合,返 `[{group, versions[], active}]` |
| T4.3 | 前端数字资产屏:同 group 多版本折叠 + 版本 tab + 切换 |
| T4.4 | `entity_set_active` API + binding 批改(只改 draft/complete) |
| T4.5 | `[+ 新版本]` 先支持"上传本地文件作新版本" |

### 二期(自动重做)— 约 1 天
| 子任务 | 内容 |
|---|---|
| T4.6 | `entity_new_version` API:单实体局部重做(调 LLM + 出图) |
| T4.7 | 版本对比 UI |

## 6. 风险

| 风险 | 应对 |
|---|---|
| logical_group 序号错位(用户改剧本) | UI 提供手动归并/拆分;序号 + 名字双重校验 |
| binding 批改影响已提交任务 | 只批改 draft/complete 状态 |
| 多版本占用 N 倍存储 | 接受;archive 老版本=隐藏不删 |
| EXTRA_CONFIG 加键与现有读取冲突 | 加键是 JSON 增字段,旧读取忽略未知键,兼容 |
